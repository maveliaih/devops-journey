# Repository Conventions

Данный документ описывает стандарты работы с репозиторием `devops-journey`.

## 1. Правила именования веток (Branch Naming)
Используем префиксы для обозначения типа задачи:
- `feat/` — новая функциональность (например, `feat/add-nginx-config`).
- `fix/` — исправление багов или опечаток.
- `docs/` — работа только с документацией.
- `chore/` — рутинные задачи (обновление зависимостей, `.gitignore`).

## 2. Стандарт коммитов (Conventional Commits)
Формат сообщения: `<type>(<scope>): <subject>`

### Типы (Types):
- **feat**: Новая фича (соответствует минорной версии).
- **fix**: Исправление ошибки (соответствует патч-версии).
- **docs**: Изменения в документации.
- **style**: Форматирование, пропущенные точки с запятой (без изменения логики).
- **refactor**: Правка кода без изменения поведения.
- **perf**: Улучшение производительности.
- **test**: Добавление или правка тестов.
- **build/ci**: Настройка сборки и CI/CD пайплайнов.
- **chore**: Обновление настроек репозитория.

### Области (Scopes):
Примеры областей для DevOps: `(git)`, `(nginx)`, `(docs)`, `(linux)`.

## 3. Правила работы с историей
- Предпочтительно использование **Merge Commit** для вливания фич в `main`, чтобы сохранить историю веток.
- Для мелких правок и синхронизации с `main` допускается **Fast-forward**.
- Важные этапы обучения фиксируются **аннотированными тегами** (`git tag -a`).

## 4. Оформление сообщений
- Заголовок: до 50 символов, инфинитив/повелительное наклонение.
- Тело (Body): объясняет **почему** внесены изменения, отделяется пустой строкой от заголовка.
